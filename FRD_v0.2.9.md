## 1. 문서 목적 및 적용 범위

### 1.1 PRD와의 관계

- **PRD**: 왜(Why)·무엇을(What) 제품 관점에서 정의한다(정책, IA 의도, 4대 기능 축, 범위 외).
- **FRD**: 각 기능이 **어떻게 동작해야 하는지(How)** 를 **검증 가능한 조건**으로 쪼개어 기술한다.
- 정책 문구(개인정보, 정산 책임 분리, 외부망 등)는 PRD와 충돌 시 **PRD를 우선**한다.

### 1.2 시스템 경계(기능 관점)

| 구분 | 파트너센터(본 화면) | 파트너센터 어드민 | 정산·마스터 백엔드 |
|------|---------------------|-------------------|---------------------|
| 역할 | 조회·다운로드·문의·분석 UI | 권한·스코프·사용자·제휴사 관리 UI | 정산 산출, 마스터 원장, API·배치 |
| **운영 어드민(레거시)** | — | AS-IS **설치형 Windows 앱** 등 — 제휴사 설치 불가 사례 존재(PR **§4**) | 일부 마스터·배치 |

- **정산 계산·확정·대사 로직**은 파트너센터 범위에 포함하지 않는다(통합 PRD §2.4 등 정본).
- 본 FRD의 **데이터 표시** 요구사항은 “정산 시스템(또는 연동 계층)이 제공하는 결과”를 전제로 한다.

## 2.5 파트너센터 진입 전제 조건

### FR-PRE01 — 브랜드 프로필 진입 게이트 **(P0)**

- **설명**: 파트너센터에 접근하기 전, 비즈니스센터에서 브랜드 프로필이 **승인(Active)** 상태여야 한다. 로그인 직후·세션 갱신 시 브랜드 프로필 상태를 확인하고, 미충족 상태에 따라 아래 흐름으로 분기한다.
- **배경**: 파트너센터는 사업자 인증을 완료한 제휴사 담당자만 이용하는 서비스다. 비즈니스센터 브랜드 프로필 심사·승인 프로세스가 해당 자격 검증을 담당하므로, 승인 완료 여부를 진입의 필수 관문으로 설정한다.

#### PRE01-1. 상태별 분기

| 브랜드 프로필 상태 | 파트너센터 처리 |
|-------------------|-----------------|
| **없음 (미생성)** | 비즈니스센터 브랜드 프로필 생성 페이지로 이동 안내 CTA 제공, 파트너센터 본 화면 진입 차단 |
| **(심사)요청 / 재심사** | 심사 대기 중 안내 화면 표시, 파트너센터 본 화면 진입 차단 |
| **반려** | 반려 사유 안내 + 비즈니스센터 재심사 신청 페이지 링크 제공, 진입 차단 |
| **승인** | 파트너센터 정상 진입 허용 |
| **삭제 대기 / 삭제 완료** | 브랜드 프로필 재생성 안내 및 비즈니스센터 이동 CTA 제공, 진입 차단 |
| **이용 제한** | 이용 제한 안내 및 문의 채널 안내, 진입 차단 |

#### PRE01-2. 수용 기준

- 브랜드 프로필 상태 확인은 **로그인 직후** 및 **세션 갱신 시** 수행한다.
- 상태 확인 API 장애 시 사용자에게 일시적 오류 안내를 제공하고, 재시도 방법을 안내한다.
- 차단 화면에서 **비즈니스센터로 이동하는 명확한 CTA**를 제공한다(이동 URL은 비즈니스센터 연동 명세에서 확정).
- 브랜드 프로필 삭제(FR-BP06) 또는 이용 제한(FR-BP07) 발생 시, 파트너센터 세션·접근 권한을 **즉시 회수**한다.
- 파트너센터 어드민(FR-AD03)의 권한 요청 시 "비즈센터 가입 여부 검증"(AD03-7·AD03-11)과 동일 흐름에서 브랜드 프로필 승인 여부도 확인한다.

#### PRE01-3. 참조

- 상태 정의 및 매트릭스: `FRD_비즈니스센터_v0.1.0.md` **FR-BP03** §BP03-1~BP03-2

- 이용 제한 시 접근 차단: `FRD_비즈니스센터_v0.1.0.md` **FR-BP07**

## 3. 액터 및 권한(기능 전제)

PRD §4 이용자 구분을 기능 전제로 한다. **파트너센터 어드민의 권한 관리 도메인**에 대한 기능·데이터·운영 규칙의 정본은 **§15 FR-AD03** 이다. 권한의 **3요소(역할·범위·권한)** 와 **최종 권한 계산식**, **요청 라이프사이클**, **감사 로그**는 **§15 FR-AD03** 본문(AD03-1~AD03-18)에서 정의한다.

화면별로 다음이 **인가 결과**로 강제되어야 한다.

- **메뉴 가시성**: 역할·스코프에 없는 메뉴는 비노출 또는 비활성.
- **대시보드·고객 분석(PR §9.1)**: **제휴사 본사 담당자** 역할만 메뉴 노출 및 지표 API 호출 가능. **매장 담당자**는 **조회 불가**(비노출·차단). 노출 데이터는 **개인정보를 제외한 집계**만. **구매 적립 제휴사** 범위를 우선 적용.
- **데이터 스코프**: 제휴사·브랜드·매장 범위 밖 데이터는 조회·다운로드·API 응답에서 **차단**한다. **프론트엔드 표시 제어만으로는 불충분**하며, 서버 API에서 동일 규칙으로 강제한다(§15 **FR-AD03** AD03-12).
- **식별정보(PR §2.9.1)**: 기본 응답은 마스킹·집계; 원문·마스킹 해제·다운로드는 **권한·사유·감사** 규칙에 따름(구현 시 별도 보안 명세와 합의).

### 3.1 CJ ONE 마스킹 정책(표시 규칙)

파트너센터 화면·API 응답에 **개인정보·식별값**을 노출할 때(마스킹 해제 권한이 없는 기본 경로), 아래 **CJ ONE 마스킹 정책**을 따른다. **원문 노출·다운로드·해제**는 [§3](#3-액터-및-권한기능-전제) 및 별도 보안·감사 명세가 우선한다.

| 항목 | 규칙 | 비고 |
|------|------|------|
| **이름** | **3글자**인 경우: **성(맨 앞 1글자)만 노출**하고 나머지는 각 `*` 로 마스킹한다. **3글자가 아닌 경우**: **가운데 2글자**를 `*` 로 마스킹한다. | 2글자 이하·복수 성 등 **경계 케이스**는 보안 명세에서 보완한다. |
| **닉네임** | **앞 2글자·뒤 2글자**는 노출하고, **그 사이**를 `*` 로 마스킹한다. | **비즈니스명(상호 등)** 필드에 동일 규칙을 적용할 수 있다. |
| **아이디** | **앞 2글자**만 노출하고, **나머지 전부**를 `*` 로 마스킹한다. | |
| **휴대전화번호** | **가운데 4자리**를 `*` 로 마스킹한다(형식은 `010-****-5678` 등 화면 표준에 맞춤). | |
| **이메일** | **로컬 파트(@ 앞)**: **앞 2글자**만 노출하고 나머지는 `*` 처리. **로컬 파트 전체 길이가 2자 이하**인 경우에도 **`@` 직전에 `**` 를 추가**한다. **`@` 뒤 도메인**은 **마스킹하지 않는다**. | 예: `ab@naver.com` → `ab**@naver.com`. `abcd@naver.com` → `ab****@naver.com`. |

**수용 기준**

- 목록·상세·다운로드 미리보기 등 **동일 데이터 유형**에 동일 규칙을 적용한다(화면별 임의 변경 금지).
- 마스킹 **구현 위치**(클라이언트 표시 전용 vs 서버 응답)는 아키텍처와 합의하되, **노출 결과**는 위 표와 일치해야 한다.

## 4. 공통 기능 (Cross-cutting)

### 4.0 제휴사 정산 운영 의견(반영 매핑)

아래는 **정산 업무를 수행하는 제휴사 담당자** 의견을 기능 요건에 반영한 것이다(예: 올리브영·ONS 등). 상세 수용 기준은 각 FR 본문에 둔다.

| 주제       | 요지                                                                           | 본 FRD 위치                     |
| -------- | ---------------------------------------------------------------------------- | ---------------------------- |
| 매장 검색    | 매장코드를 외우지 않고 **매장명**으로 검색하는 경우가 많다                                           | **FR-C06** C06-11            |
| SNB 접힘   | **가로로 넓은 데이터(표)** 가 있을 때 사이드바를 접으면 본문 가로 확보에 도움이 된다                          | **FR-N01**                   |
| 총 건수     | 화면에 **총 데이터 건수**가 보이면 조회 결과 규모를 직관적으로 파악·**다운로드 행 수와 대사**하기 쉽다               | **FR-C04** C04-2, **FR-C05** |
| 페이지당 행 수 | 한 화면 **약 30행** 정도가 적당하다는 의견(다운로드 파일 행 수 제한과는 별개)                             | **FR-C04** C04-2             |
| 필터 초기화   | 별도 **「필터 초기화」** 보다, **검색 필터**의 **전체 선택 / 전체 해제**(다중 선택 항목)로 대체 가능할 수 있다(ONS) | **FR-C03** C03-4             |
| 인피니트 스크롤 | 서비스 기획 측 **인피니트 스크롤 적용 가능 여부** 검토                                            | **§18** 미결정                  |

### FR-C01 — 애플리케이션 셸·레이아웃 **(P0, Mock)**

- **설명**: 고정 사이드바·상단 헤더·본문 영역·푸터로 구성된 단일 포털 레이아웃을 제공한다.
- **페이지 스크롤(본문 영역)**
  - **헤더(타이틀 영역)**: 세로 스크롤 시 **고정**되거나, **최소 높이**를 유지해 타이틀·컨텍스트가 사라지지 않도록 한다.
  - **탭**: 본문이 길어질 때 **sticky** 로 두어 **항상 상단**에 고정된다(탭 아래 콘텐츠만 스크롤되는 패턴).
- **수용 기준**
  - 현재 활성 메뉴에 맞춰 헤더 **제목·설명**이 일치한다.
  - 푸터에 서비스 안내·약관 링크 등은 **정책 확정 후** 반영 가능하도록 확장 가능하다(PR §6.1).

### FR-C02 — 브랜드·매장 컨텍스트 선택 **(P0, Mock)**

- **설명**: 브랜드·매장 **계층(직영/가맹 등)** 기반 다중 선택으로 이후 화면의 조회 범위를 결정한다(PR §6.1, §8.1). PR **§8**: **미사용·휴면 등 정책상 제외되는 제휴사**는 파트너 **본화면** 선택 목록에서 **기본 비노출**한다(정책 플래그·마스터와 연동).
- **수용 기준**
  - 선택 값이 변경되면 **동일 세션 내** 연동 화면의 필터·집계 기준이 일관되게 반영된다(목업: `BrandStoreContext`).
  - 실서비스: 선택 가능 조합은 **권한 스코프**와 교차 검증된다.
  - 다중 선택 UI는 **FR-C03 C03-4**(전체 선택·전체 해제·필터 초기화와의 관계) 및 **FR-C10**(다이얼로그·적용 버튼)과 맞춘다.

### FR-C03 — 기간·정산월 필터 패턴 **(P0, Mock)**

- **설명**: 정산·원장 화면에서 정산월, 일자 범위 등 **공통 필터 컴포넌트**를 사용한다.

#### C03-1. 디폴트·정산월·기간 매핑

| 개념             | 정의                                                                  |
| -------------- | ------------------------------------------------------------------- |
| **디폴트**        | **지난달 ~ 지난달**(시작·종료 모두 전월)                                          |
| **정산월**        | **전월 ~ 전월**(단일 기준월에 대응)                                             |
| **실제 조회 기간**   | **전월 1일 ~ 전월 말일**                                                   |
| **일자 직접 선택 시** | 사용자가 고른 **일자가 속한 기준월**이 **정산월(입력값)** 으로 반영된다(해당 월 1일~말일로 내부 매핑 가능). |

#### C03-2. 기본 모드: 「정산월 선택」 단일화

- **UI**: **YYYY.MM** 단일 선택(월 단위 피커 등).
- **디폴트**: **전월**.
- **내부 처리**: 표시는 월 단위이나, 조회 API·집계에는 **해당 월 1일 ~ 해당 월 말일**로 자동 매핑한다.

#### C03-3. 확장 모드: 「직접 기간 선택」

- **UI**: **시작일 ~ 종료일**을 달력 등으로 직접 선택한다.
- 기본 모드와 전환(토글·고급 펼침 등) 여부는 화면별로 정하되, **동일 화면에서 두 모드의 의미가 중복되지 않게** 라벨·도움말을 구분한다.

#### C03-4. 필터 초기화 vs 전체 선택·해제(제휴사 운영 의견)

- **ONS 정산 담당자 의견**: 단일 **「필터 초기화」** 기능 대신, **검색 필터** 영역에서 다중 선택 항목에 대해 **전체 선택**·**전체 해제**(또는 동등한 일괄 토글)를 제공하는 방식으로 **동일 니즈를 충족**할 수 있다.
- **요구**
  - **다중 선택 필터**(브랜드·매장·거래 유형 등)가 있는 화면에서는, 해당 그룹에 **전체 선택**·**전체 해제**를 두어 한 번에 범위를 넓히거나 좁힐 수 있게 한다(세부 레이블은 화면별).
  - **「필터 초기화」**(모든 필터·기간을 디폴트로 되돌림)는 **필수가 아니다**. 전체 선택·해제와 **기간·정산월**의 명확한 디폴트(C03-1~C03-3)로 대체 가능하면 **중복 버튼을 두지 않는다**.
  - 기간·검색어까지 한 번에 초기화해야 하는 **강한 리셋**이 필요한 화면에만 **초기화**를 별도 검토한다.
  - **페이지 전역 초기화**(C03-6)가 있는 화면에서는, 그 버튼이 **해당 페이지 조회 결과에 영향을 주는 모든 조건**을 디폴트로 되돌린다는 점과 역할이 겹치지 않게 라벨·배치를 구분한다.

#### C03-5. Datepicker(기간·일자 선택) 동작

- **연·월 이동**: 달력 UI에서 **연도·월 단위 이동** 컨트롤(이전/다음 월, 연도 드롭다운 등)을 제공해 **빠른 기간 탐색**이 가능하다.
- **데이터 호출 시점**: 일자·범위를 달력에서 바꿀 때마다 API를 호출하지 않고, 사용자가 **「확인」**(또는 동등한 최종 확정) 버튼을 눌렀을 때만 **조회·데이터 호출**을 수행한다(미확정 상태의 중간 선택으로는 목록이 갱신되지 않음). **키워드 검색**의 debounce·Enter 동작은 **FR-C06**이며 본 절과 혼동하지 않는다.
- **수용 기준**: `확인` 전 취소·닫기 시 **기존 조회 조건·결과**를 유지한다.

#### C03-6. 페이지 전역 「초기화」 버튼

- **정의**: 페이지 상단 등 **전역**에 배치된 **「초기화」** 버튼을 클릭하면, **해당 페이지의 데이터 결과에 영향을 주는 모든 조건**을 **각 항목의 디폴트**로 되돌린다.
- **포함 대상(예시)**: 정산월·기간·직접 기간 모드, 브랜드·매장·다중 선택 필터, 키워드 검색, 정렬(화면별 정의 시), 페이지 번호 등 — 화면별로 **초기화 대상 목록**을 명세에 적어 둔다.
- **제외·주의**: **글로벌 브랜드·매장 컨텍스트(FR-C02)** 가 페이지 밖에서 결정되는 경우, 전역 초기화가 그 값까지 되돌리는지는 **IA 정책**으로 정하고 FR에 명시한다.
- **C03-4와의 관계**: 다중 선택 필터의 **전체 선택·전체 해제**는 **부분 범위 조정**용으로 유지할 수 있다. **전역 초기화**는 **한 번에 디폴트 복귀**를 보장하는 별도 수단이다.

- **수용 기준**
  - 잘못된 기간(시작 > 종료 등) 입력 시 제출 전 검증 또는 명확한 오류 메시지를 제공한다.
  - 조회 시 로딩·빈 결과·오류 상태를 구분해 표시한다(실서비스).

### FR-C04 — 표·정렬·페이지네이션 **(P0, Mock)**

- **설명**: 대량 데이터는 표 형태로 제공하고, 필요 시 정렬·페이지 이동·건수 표시를 지원한다.

#### C04-1. 스크롤·총계 행

- **세로 스크롤**: 데이터 영역만 스크롤한다. **컬럼 헤더(표 헤더 행)** 와 **총계 행**은 스크롤 대상에서 **제외**하고 항상 식별 가능하게 둔다(고정 헤더·고정 풋터 패턴).
- **가로 스크롤**: **고정(pinned) 컬럼**은 가로 스크롤 시에도 **제외**하고 화면에 고정한다(예: 키·매장명 등). 스크롤되는 영역은 고정 컬럼 오른쪽의 가변 컬럼만 해당한다.
- **총계 행**: 표 **하단**에 노출한다(데이터 본문 아래, 스크롤 영역 밖에 두는 것이 원칙 — C04-1 세로 규칙과 일치).

#### C04-2. 총 건수·페이지 크기·스크롤 방식

- **총 데이터 건수 노출**
  - 조건(기간·검색·필터·스코프)에 맞는 **전체 건수**를 화면에 **명시**한다(예: 「총 N건」). 사용자가 **목록에 보이는 행 수만으로 전체 규모를 오해하지 않도록** 한다.
  - **엑셀 등 다운로드** 후 파일 **행 수**와 화면 **총 건수**를 비교해 **부분 다운로드·오류·누락 여부**를 가늠할 수 있게 한다(정밀 대사는 백엔드·파일 명세와 합의 — **FR-C05**).
- **한 화면에 노출되는 행 수**
  - 정산·원장형 목록에서는 **약 30행 전후**를 **기본 페이지 크기** 후보로 둔다(제휴사 운영 의견). 화면별·밀도 설정으로 조정 가능하다.
  - **다운로드 파일**의 행 수는 **조회 조건 전체**를 담는 것이 원칙이며, **한 화면 노출 행 수와 무관**하다(대용량은 비동기·분할 — **NFR-06**, FR-C05).
- **페이지네이션 vs 인피니트 스크롤**
  - **페이지 번호 방식**과 **인피니트 스크롤**(스크롤 하단에서 추가 로드) 중 선택은 **UX·접근성·성능·구현 비용**을 검토해 확정한다(§18). 본 문서에서는 **서버 사이드 페이징·총 건수 표기**를 전제로 한다.

- **수용 기준**
  - 기본 정렬 키·정렬 방향이 화면별로 정의되어 있거나 사용자가 선택할 수 있다.
  - “조회 결과 없음”과 “오류”를 구분한다.
  - **검색(FR-C06)** 과 병행 시, 검색어 변경 시 **현재 페이지는 1페이지로 리셋**되고 결과 집합이 **전체 교체(replace)** 된다(FR-C06, C06-6). 인피니트 스크롤 채택 시에도 **검색어 변경 시 목록 초기화·맨 위로 스크롤** 등 동등한 동작을 정의한다.

### FR-C05 — 다운로드·다운로드 이력 **(P0, P1)**

- **설명**: 공문·엑셀 등 파일 다운로드 UI 및 **다운로드 이력** 조회를 지원한다(PR §6.4). PR **§5.1**: 적립·사용 **건별 대사**용 **대용량 내역 파일** 제공, **정산 대사 파일 종류 정리**, 저장소 **AWS 이전** 시 제휴사 연동 변경 — **NFR-06**과 함께 설계한다.
- **총 건수와 다운로드 행 수(운영 의견 반영)**: 화면에 표시된 **총 데이터 건수**(FR-C04 C04-2)와 사용자가 받은 파일의 **데이터 행 수**가 **의도적으로 다를 수 있는 경우**(부분 샘플·비동기 분할·제한 등)에는 **다운로드 완료 안내·파일 내 건수·총 건수 관계**를 명확히 한다. 사용자가 「화면 1,000건인데 파일 100행만 있다」처럼 **오류 여부를 스스로 판단**할 수 있도록 한다(세부는 파일 포맷·배치 명세).
- **수용 기준**
  - 권한 없는 유형의 다운로드는 수행할 수 없다.
  - 정산 리포트의 **비밀번호 생성·변경**, 재다운로드·유효기간 등은 통합 PRD §2.10 및 보안 설계 확정 후 반영 **(P1)**.

### FR-C06 — 검색 필드(키워드 검색) **(P0, Mock)**

검색 필드는 단순 입력 UI가 아니라 **조회 효율 개선을 위한 핵심 컴포넌트**로 정의한다. 대량 데이터 환경에서 특정 조건을 빠르게 탐색하고, **엑셀 다운로드 후 가공에 대한 의존**을 줄이며 정산·조회 업무 시간을 단축하는 것이 목표다.

#### C06-1. 목적 (Why)

- **해결 과제**: 메뉴만으로는 목표 데이터 도달이 어렵고, 필터만으로는 원하는 건을 좁히기 비효율적인 경우가 있다.
- **기대 효과**
  - 검색 후 **즉시** 화면에서 결과 확인
  - 엑셀 다운로드 의존도 감소
  - 정산·원장 조회 업무 시간 단축

#### C06-2. 적용 범위

- **헤더 글로벌 검색**(PR §6.1) 및 **적립/사용 내역·제휴 정산·기프트 등 목록형 화면**에 동일 패턴의 검색을 적용할 수 있다.
- 화면별로 검색 대상 필드·placeholder 문구는 **해당 업무에 맞게** 정의하되, **동작·검증·성능 규칙은 본 절을 따른다**.
- **회원번호 등 식별정보**가 검색 대상에 포함되는 경우, 조회·표시는 **§3**·**§3.1** 및 PR **§2.9.1** **마스킹·권한** 규칙을 반드시 준수한다.

#### C06-3. 기본 동작 (Core behavior)

| 항목 | 요구 |
|------|------|
| 입력 UI | **단일 라인** 텍스트 필드 |
| Placeholder | 해당 화면에서 **검색 가능한 항목**을 명시 (예: 「매장명 / 승인번호 / 회원번호 입력」) |
| 검색 트리거 | (1) 입력 변경 후 **debounce 300~500ms** 경과 시 자동 검색 (2) **Enter** 입력 시 **즉시** 검색(강제) (3) **검색 버튼**은 optional — 모바일·터치 환경에서는 유지 권장 |
| 조회 버튼 의존 | 목록의 **기본 데이터는 검색어 없이도** 진입 시 자동 조회(또는 preload)된다. 사용자가 별도 「조회」 버튼 없이도 **초기 리스트**를 볼 수 있어야 한다. |
| 초기 상태 | 화면 진입 시 **검색어 비어 있음** 상태에서 **전체(또는 필터만 적용된) 목록**이 로드된다. |

#### C06-4. 입력 제약 (Validation)

| 항목 | 요구 |
|------|------|
| 최대 길이 | 기본 **50자** hard limit(초과 입력 불가). ID·번호 중심 화면은 **30자** 권장(화면별 명세에서 확정). |
| 문자 종류 | 일반 검색: 한글·영문·숫자 허용. **승인번호 등 숫자 전용** 의도 필드는 **숫자만** 허용하도록 화면별 정의 가능. |
| 특수문자 | SQL injection 등 방지를 위해 **허용 문자 집합 제한** 또는 서버·게이트웨이 단 검증(보안 명세와 합의). |
| 공백 | **앞뒤 trim**. 문자열 **중간 공백**은 유지(매장명 등). |

#### C06-5. UX 인터랙션

| 항목 | 요구 |
|------|------|
| Clear | 입력값이 있을 때 필드 우측 **X(지우기)** 노출. 클릭 시 입력 초기화 후 **검색어 없음 기준으로 전체 목록 재조회**. |
| 포커스 | 포커스 시 **최근 검색어** 또는 **추천 검색어** 노출 가능(P1, C06-8과 연계). **blur** 후에도 **결과 목록은 유지**. |
| 피드백 | 요청 중 **로딩** 표시. 결과 0건 시 **empty state**: 「검색 결과 없음」+ **검색 조건 가이드**(어떤 값을 넣을 수 있는지). |

#### C06-6. 검색 결과 정책 (Result handling)

| 항목 | 요구 |
|------|------|
| 갱신 방식 | 검색어가 바뀌면 이전 결과를 **완전히 대체**한다. **페이지네이션은 1페이지로 리셋**. |
| 매칭 규칙 | 텍스트 필드는 **부분 일치(contains)** 를 기본으로 한다(완전 일치만 강제하지 않음). **숫자형 식별값**(승인번호 등)은 **prefix match 우선** 정책을 적용할 수 있다(API 명세에서 확정). |
| 검색 대상 필드(기본 가이드) | **필수 후보**: 승인번호, 매장명, 거래ID. **선택 후보**: 회원번호, 제휴사명. 화면별로 실제 인덱스·노출 컬럼과 맞춘다. **업무 우선순위**는 고객 프로필 탐색보다 **거래·포인트 흐름** 추적에 맞춘다. |

#### C06-7. 성능 및 제한

| 항목 | 요구 |
|------|------|
| Debounce | **300~500ms**(과도한 API 호출 방지). |
| 페이지 크기 | **정산·원장형 목록**: 한 번에 **약 30건**을 기본 후보로 둔다(제휴사 운영 의견, FR-C04 C04-2와 정합). **기타 목록**: **50~100건** 범위에서 화면별 정의. 추가 로드는 **pagination** 또는 **infinite scroll**(§18 검토). |
| 실행 위치 | **서버 사이드 검색 필수**. 대량 원장을 **클라이언트 전체 로드 후 필터링**하는 방식은 사용하지 않는다. API는 **검색어·필터·스코프·페이지**를 파라미터로 받는다. |

#### C06-11. 매장코드·매장명 검색(제휴사 운영 의견)

- **현장 패턴**: 담당자가 **매장코드를 외우지 않는 경우가 많고**, **매장명**으로 검색하는 빈도가 높다.
- **요구**
  - 매장과 연관된 검색 UI에서는 Placeholder·도움말에 **매장명 검색이 가능함**을 **매장코드와 동등 이상으로** 드러낸다(예: 「매장명 / 매장코드」 순서·병기).
  - API·인덱스 설계 시 **매장명 부분 일치(contains)** 등 실사용 패턴을 반영한다(C06-6 매칭 규칙과 합치).
  - **매장코드 전용** 입력이 필요한 화면이더라도, 가능하면 **이름 검색·자동완성**(C06-8)으로 보완하는 방향을 검토한다.

#### C06-8. 확장 (Advanced)

| 항목          | 태그  | 요구                                                              |
| ----------- | --- | --------------------------------------------------------------- |
| 필터 결합       | P0  | **검색어 + 기간·거래유형·브랜드/매장 등** 화면 필터를 **동시에** 적용. API에 모든 조건을 전달한다. |
| 자동완성·최근 검색어 | P1  | 매장명·제휴사명 등 **자동완성**, **최근 검색어** 저장(개인정보·보관 기간은 정책 합의).          |
| 키워드 하이라이트   | P1  | 결과 목록에서 **매칭 구간** 시각적 강조.                                       |

#### C06-9. 예외 처리 (Edge cases)

- 허용되지 않는 형식·문자: **검증 메시지** 표시, 검색 요청 생략 또는 서버 오류 메시지 표시.
- 결과 0건: **가이드가 포함된 empty state**(C06-5).
- 네트워크 실패: **재시도** 가능한 UI(버튼 또는 자동 재시도 정책은 구현 가이드에 따름).

#### C06-10. 파트너센터 맥락 차별 포인트 (요약)

1. **조회 전용 「검색」 버튼에 의존하지 않는** 즉시 검색 구조(debounce + Enter).
2. **대량 데이터**: 서버 검색 + 페이지 크기 제한.
3. **정산·거래 중심** 검색 대상 정의(식별정보 검색은 권한·마스킹과 세트).
4. **엑셀 의존도 감소**를 명시적 목표로 둔다.

**수용 기준(통합)**

- 위 C06-3~C06-9를 만족하는 **공통 검색 컴포넌트 또는 동등 규약**을 두고, 목록 화면에서 재사용할 수 있다.
- 목업(`Header` 등)은 현재 UI 자리만 제공하며, 실서비스 연동 시 본 요구사항을 충족해야 한다.

#### C08-1. 가이드 콘텐츠 채널(GitBook)

#### C08-2. 페이지별 가이드(인앱)

- **형태**: 화면별 **도움말 아이콘·링크**에서 열리는 **다이얼로그**(또는 드로어)로 제공한다.
- **콘텐츠 유형**
  - **텍스트 팝업형**: 정책·절차 요약 등 **정적 텍스트**(마크다운 등) 삽입.
  - **유튜브 영상형**: 해당 화면 조작을 설명하는 **외부 영상 임베드**(iframe 등).
- **구현 전제**: 텍스트·영상 **동시 제공**, 다국어, 접근성(자막·키보드), 외부망·쿠키 정책에 따른 **실제 제작 가능 여부·개발 공수**는 **기획·보안·개발 합의 후** 확정한다(§18).

- **수용 기준**
  - 가이드가 없는 화면에서는 진입점을 노출하지 않거나 비활성 처리한다(화면별 명세).

### FR-C09 — 헤더 알림 **(P1, Later)**

- **설명**: 알림 벨·알림 목록 등. **승인 대기·권한 요청 대기** 등 운영 건에 대한 **별도 알림**(인앱·이메일 등)은 PR **§3.1** 과제 해소를 위해 우선순위화한다. **헤더 검색 입력은 FR-C06**을 따른다.
- **수용 기준**: 실데이터·정책 확정 후 동작 정의. 목업은 UI 자리만 제공.

### FR-C10 — 멀티 선택 다이얼로그 **(P0, Mock)**

- **설명**: 브랜드·매장·유형 등 **다중 선택**을 모달·시트 등 **다이얼로그**로 제공하는 공통 패턴이다(FR-C02·정산 필터 등과 연계).
- **요구**
  - 다이얼로그 하단(또는 고정 푸터)에 **「적용」** 버튼을 둔다. 사용자가 목록에서 체크·해제를 변경한 뒤 **「적용」** 을 눌렀을 때만 **부모 화면의 필터 값이 확정**되고, 필요 시 **조회 API**가 호출된다(datepicker의 **확인 후 조회**와 동일 원칙).
  - **취소**·배경 클릭 닫기 시: **열기 전 상태**로 되돌린다(미적용 변경 폐기).
- **수용 기준**
  - **전체 선택·전체 해제**(FR-C03 C03-4)가 다이얼로그 내에 있으면, 그 변경도 **「적용」** 전까지는 부모에 반영되지 않는다(일관성).

### FR-AD03 — 권한 관리(1차 핵심) **(P0, Mock)**

권한 관리는 **파트너센터 어드민의 핵심 도메인**으로, 아래 **3개 하위 기능**으로 구성한다. 메뉴·화면 책임은 분리하되, **운영 흐름에서는 상호 연결**되어야 한다.

| 하위 기능 | 책임(요약) | 연동 요구 |
|-----------|------------|-----------|
| **사용자 관리** | 계정·권한 상태 조회·변경 | 사용자 상세에서 **현재 Role·권한(Permission)·Scope** 조회 가능 |
| **권한 관리** | Role·Permission·Scope 정책 정의 | 권한 매트릭스·마스터 데이터의 정본 |
| **권한 요청 관리** | 신청 건 검토·승인·반려 | 승인 시 **사용자 권한에 즉시 반영**(동기화 또는 확정 시점 규약 준수) |

PR §7·§8.2, PR **§3.3~3.4**(가입 전 요청 가드·검증·일괄 입력)와 정합한다. 입력 UX 공통 규칙은 **FR-C06**(trim·debounce 등)과 맞춘다.

#### AD03-1. 기능 범위

- 상기 3 하위 기능은 **각각 독립 메뉴·화면**으로 둔다.
- **사용자 상세** ↔ **권한 요청·권한 마스터** 간 이동·상호 참조가 가능해야 한다(예: 상세에서 요청 이력·현재 부여 결과 확인).

#### AD03-2. 권한 모델(RBAC + Permission 커스터마이징 + Scope)

- 구조는 **RBAC 기반**이나, 단순 Role 부여로 끝내지 않고 **Permission 커스터마이징**과 **Scope(범위)** 를 함께 관리한다.
- **최종 권한 구조 3요소**: **역할**(누구인가) · **범위**(어디까지 볼 수 있는가) · **권한**(무엇을 할 수 있는가).
- 시스템은 사용자별로 아래를 **저장·조회**할 수 있어야 한다.
  - **기본 Role**
  - **Role에 매핑된 기본 Permission**
  - **사용자별 추가 Permission**
  - **사용자별 제거 Permission**
  - **최종 Scope 범위**
- **최종 권한 계산식**: `최종 권한 = Role 기본 권한 + 개별 추가 권한 − 개별 제거 권한`
- **Role 없이 Permission만** 가진 사용자를 허용한다. 이 경우 **인트로·정보 수정·FAQ** 등 **제한적 공통 기능**만 접근 가능한 **경량 사용자**로 본다.

#### AD03-3. 역할 정의

- **초기 표준 Role(1차 기준)**: **본사 관리자**, **매장 관리자**, **정산 담당자**.
- **후속 확장 후보**: 영업, IT, 재무/회계, 마케팅 등 — 조직별 편차가 크므로 **1차 오픈에서는 표준 Role 수를 최소화**한다.
- **방향**: **본사 관리자·정산 담당자 계열**은 Role 수를 늘리기보다 **Permission 커스터마이징**으로 보완. **매장 권한**은 단순화, **본사 권한**은 세분화 필요(현장 인터뷰 반영).

#### AD03-4. 권한 범위(계층형 Scope)

- Scope는 **제휴사 > 브랜드 > 매장** 계층으로 설계한다.
- **상위 단위 부여 시 하위 범위 자동 포함**(예: 제휴사 단위 권한 → 해당 제휴사 하위 브랜드·매장 모두).
- 사용자에게는 아래 **3개 중 하나를 필수**로 선택하게 한다: **제휴사 단위** · **브랜드 단위** · **매장 단위**.
- 각 단위 내에서는 **All 또는 부분 선택**이 가능해야 한다.
- **용도 가이드**: 제휴사 단위 — 제휴사 마스터·정산 담당자 수준; 브랜드 단위 — 브랜드별 마스터; 매장 단위 — 매장 운영 중심.
- **대시보드**: **본사 관리자만** 조회 가능(PR §9.1·본 FRD §3와 일치). Scope에 포함되지 않은 데이터는 **조회·다운로드·API 응답** 모두 차단한다.

#### AD03-5. 권한 종류(기능 행위 기준)

- 권한은 **메뉴명이 아니라 기능 행위** 기준으로 정의한다. 상위 유형 예: **조회/다운로드**, **승인/반려**, **설정/관리**.
- **설정/관리** 등은 시스템에서 **더 세분화**할 수 있어야 한다(예: 사용자 관리, Role 설정, Scope 설정, 요청 승인 정책 관리 등 내부 분리).

#### AD03-6. 권한 운영 전략(저복잡도 / 고복잡도)

| 구분 | 대상 | 운영 방식 |
|------|------|-----------|
| **저복잡도** | 매장 사용자 | **단일 Role** 중심, **자동 Scope**, 신청 후 **일괄 승인** 가능 |
| **고복잡도** | 본사 사용자·정산 담당자 | **Role + Permission 커스터마이징**, 승인자 **수동 검토** 후 부여 |

(현장 인터뷰의 **투트랙 운영** 제안과 정합.)

#### AD03-7. 권한 요청 라이프사이클

요청은 아래 **상태 흐름**을 지원해야 한다: **요청 → 검증 → 승인 → 변경 → 회수 → 만료**.

- 사용자는 **가입 또는 권한 변경 시** Role을 선택해 요청할 수 있어야 한다.
- **가입 여부를 먼저 검증**한다. **미가입** 사용자는 **비즈센터 가입 후** 요청하도록 **UI·서버 가드**(현행 오류 방지, PRD·FRD 방향과 일치).
- **브랜드 프로필 승인 여부**도 동일 검증 흐름에서 확인한다. 브랜드 프로필이 없거나 승인 상태가 아닌 사용자는 파트너센터 권한 요청 전에 비즈니스센터에서 브랜드 프로필 승인을 먼저 완료하도록 안내한다(**FR-PRE01** 정본, `FRD_비즈니스센터_v0.1.0.md` **FR-BP03** 상태 매트릭스).

#### AD03-8. 사용자 관리 화면 요건

필수 기능:

- 사용자 **리스트 조회**(이름, 이메일, 조직, Role, Scope, **계정 상태**, **권한 상태**, 마지막 활동일 등).
- **검색·필터**, **row 클릭 시 상세 패널**.
- 상세에서 **Role**, **Permission override**, **Scope**, **이력** 확인.
- **계정 상태 변경**, **권한 회수**, **일괄 처리**.

**계정 상태**와 **권한 상태**는 **분리 저장·분리 표시**한다(로그인 가능 여부 vs 업무 권한 보유 여부).

향후 확장 시 **마지막 활동일**, **2단계 인증 여부**, **외부 소속 여부** 등 운영 정보를 함께 제공하는 방향이 적합하다.

#### AD03-9. 권한 관리 화면 요건

필수 관리 대상:

- **Role master**
- **Role–Permission 매핑**
- **사용자별 override permission**
- **Scope 정책**
- **권한 매트릭스 조회**

**권한 매트릭스**는 최소 다음 축을 지원한다: **메뉴**, **화면**, **팝업**, **버튼**, **기능**, **데이터 범위**. 운영자가 특정 Role의 **가시 메뉴·클릭 가능 버튼·조회 데이터 범위**를 **표 형태**로 한눈에 확인할 수 있어야 한다(인터뷰 요구 반영).

#### AD03-10. 권한 요청 관리 화면 요건

필수 기능:

- 요청 **리스트 조회**, **상태별 필터**
- 요청자 정보, 요청 **Role·Scope**, **검증 결과** 확인
- **개별 승인·일괄 승인**, **반려 및 반려 사유** 입력
- **승인 후 사용자 권한 자동 반영**

매장 사용자 등 **반복·단순 요청**은 **일괄 승인**이 가능해야 한다(PR: 매장 관리자 권한 요청 다건·워크플로·일괄 입력·검증·템플릿).

#### AD03-11. 권한 부여 전 검증

부여 전 검증은 **필수**이며, 최소 다음을 제공한다.

- **가입 여부** 검증
- **CJ ONE 아이디 전달 여부** 검증
- **중복 권한** 검증
- **ID 형식** 검증
- **소속** 검증 — 특히 **매장 사용자**는 **실제 해당 매장 직원 여부** 검증 절차(인터뷰 강조 사항).

#### AD03-12. 권한 기반 UI 제어

최종 권한 결과를 기준으로 **UI와 데이터 접근**을 모두 제어한다.

- 메뉴 비노출, 버튼 비활성, 화면 진입 차단, 팝업 기능 제한
- **데이터 필터링**(Scope 기준) — **프론트만으로 불충분**, **서버 API 동일 규칙 차단**

#### AD03-13. 권한 변경 이력·감사 로그

권한 변경은 **Audit log**로 저장한다. 최소 항목:

- **누가**, **누구에게**, **언제**, **무엇을**, **어떤 Role로**, **어떤 Scope로**, **부여/변경/회수**, **사유**

**N년간 보관**(정책 값은 보안·법무와 합의), **검색·필터** 가능해야 한다.

#### AD03-14. 권한 관리 영역 표 공통 요건

목록성 화면은 **FR-C04**(고정 헤더·총계 행·고정 컬럼 등)와 함께 아래를 적용한다.

- 좌측부터 **N개 컬럼 고정(pinned)** 가능
- **row 클릭 → 상세 패널**
- **검색·필터·정렬·페이지네이션**(대량 데이터 기준)
- **사용자·요청 목록**에는 **총 건수** 표시 권장

#### AD03-15. 레이아웃(사이드바 숨김)

- 사이드바 숨김·접기는 **편의 기능**으로 둘 수 있으나 **권한 관리 필수 요건은 아님**. 정산 담당자 중심 운영에서는 우선순위 낮음 — **FR-N01**과 정합.

#### AD03-16. 화면 구조 제안(IA 참고)

**권한 관리(그룹)** 하위:

1. **사용자 관리** — 사용자 목록 · 사용자 상세 패널  
2. **권한 관리** — Role 목록 · Role 상세 · Permission 설정 · Scope 정책 · 권한 매트릭스  
3. **권한 요청 관리** — 요청 목록 · 요청 상세 패널 · 검증 결과 · 승인/반려 처리  

#### AD03-17. 통합 요건 문장(도메인 정의)

권한 관리는 파트너센터 어드민의 핵심 도메인으로서, **사용자 관리·권한 관리·권한 요청 관리** 3개 하위 기능으로 구성한다. 권한 구조는 **RBAC 기반 Role + Permission 커스터마이징**을 적용하며, **역할·범위·권한**을 분리 관리한다. 사용자는 **제휴사/브랜드/매장 중 하나의 Scope 단위**를 필수로 가지며, 상위 Scope 부여 시 하위 범위를 자동 상속한다. 최종 권한은 **Role 기본 권한**과 **사용자별 추가·제거 Permission**의 조합으로 계산한다. 매장 사용자는 **단일 Role + 자동 Scope + 일괄 승인**, 본사 관리자 및 정산 담당자는 **Role + Permission 커스터마이징 + 수동 검토**를 적용한다. 시스템은 **요청·검증·승인·변경·회수·만료** 전 과정을 관리하며, **메뉴 비노출·버튼 비활성·데이터 필터링·API 차단**까지 포함한 권한 기반 UI 제어를 수행한다. 모든 권한 변경은 **감사 로그**로 저장하고 **N년간** 보관한다.

#### AD03-18. ONS 정책 담당자 문의사항(기능·정책 연계 메모)

아래는 운영·정책 확정이 필요한 **문의·전제**이며, 구현 시 IAM·인사·ONS 운영 프로세스와 합의한다.

1. **Role 저장·모니터링**  
   - 역할 그룹별 권한 조회, 사용자에게 역할 그룹 부여 등 **감사·리포팅 요구**와 연계해 **Role 및 부여 이력 저장**이 필요한지 검토한다.  
2. **소속 정보 확인 시점**  
   - 역할 그룹 대비 사용자 소속 확인, **계열사 이동**(제일제당 인사 담당자 조회 등), **매장 변경**(가입 후 ONS 정책에서 매장 정보 변경·담당자 요청 흐름), **역할 변경** 시 재검증 필요 여부.  
3. **90일 미사용 탈퇴**  
   - 보안 측과 **계정 휴면·탈퇴·권한 회수** 연계 규칙 확정(본 FRD **§18** 미결정 과제와 중첩).  
4. **탈퇴 후 과거 권한 조회**  
   - 탈퇴(예: 90일 경과) 후에도 **이름 검색 등으로 과거 부여 권한**을 확인해야 하는 운영 케이스가 있음 — **감사 로그·스냅샷 보존 기간**과 UI 검색 범위를 정의한다.

- **수용 기준**
  - AD03-1~AD03-17의 요구가 백엔드·API·UI에 반영 가능한 수준으로 추적 ID(화면별 세부 명세)에 분해된다.
  - 변경 사항 저장·승인 시 **권한 없는 호출자**는 거부된다(실서비스).
  - **감사 로그**는 AD03-13을 충족하며, 보존 연수·개인정보 최소화는 보안·법무 정책이 우선한다.

## 18. 미결정·후속 과제 (기능 관점)

- **페이지별 가이드 다이얼로그**(FR-C08 C08-2): **유튜브 임베드**·외부 스크립트·자막·**CSP·외부망** 정책 하에서의 **제작 가능 여부·유지보수 공수**; 텍스트-only 대안.
- 정산·원장 **목록 UI**: **페이지 번호** 대신 **인피니트 스크롤**(또는 혼합) 적용 여부 — **약 30행 노출**·총 건수 표기(FR-C04)·접근성(포커스 이동, 스크린리더)·성능(API 호출 패턴)과 함께 **서비스 기획·개발 합의**로 확정한다.
